# actflow — спецификация

[English](SPEC.md)

Библиотека исполнения графов задач. Граф описывает, какие задачи выполняются и куда передают результаты; исполнитель прогоняет его, передавая данные между узлами. Поддерживает синхронный и асинхронный режим, в том числе удалённые и долгие задачи.

## 1. Базовая идея

Задача только считает и возвращает результат — куда его направить, задаётся связями графа, а не самой задачей. Решение о запуске принимает исполнитель. Это даёт ему полный контроль над порядком, параллельностью и остановкой.

Модель близка к функциональному стилю и к вычислительному графу: по рёбрам текут неизменяемые данные, узлы не делят общее изменяемое состояние.

## 2. Два уровня по доступу ко времени

Всё в узле делится на два уровня. Делёж — по тому, виден ли ход времени.

**Временной уровень — тело (execute).** Чистая функция одного такта: получило значения, вернуло значения. Прошлого и будущего не видит, между тактами ходить не может, состояние не хранит.

**Вневременной уровень — контроллеры.** Видят все очереди узла и накопленный выход, держат состояние между тактами. Сюда выносится всё, что требует памяти через время: готовность, выбор кандидата, порядок, сборка батча, разборка результата.

## 3. Пакет данных

По рёбрам течёт пакет — конверт над значением.

```
пакет = значение + ярлык источника
```

**Значение** — то, с чем работает тело. **Ярлык источника** — адрес слота: куда пакет ляжет в узле-получателе. Пакет неизменяем: один результат можно направить в несколько узлов без копий.

Метаданные (порядковый номер для синхронизации и пр.) по графу не ездят — они живут в `Collected.mark`, который контроллер ввода производит в `collect()` и передаёт контроллеру вывода в `emit()` в рамках одного такта (см. раздел 6).

## 4. Ярлык источника и маршрутизация

Слот получателя определяется **ярлыком источника**, а не позицией прихода.

Узел по умолчанию помечает результат ярлыком своего источника: задача с `label="A"` выдаёт данные с ярлыком `A`. Узел с `execute(self, A, C)` кладёт пришедший `A` в слот `A`, пришедший `C` — в слот `C`.

**Разрешение ярлыка** (первое непустое): class var `type_label` → `out_labels[0]` → аргумент конструктора `label` → имя класса. Для простого графа с одним узлом на класс имена классов уникальны и явной настройки не нужно.

Если два узла одного класса отправляют в один коллектор — назовите их:

```python
w1 = Worker(label="w1")()
w2 = Worker(label="w2")()

class Collector(Task):
    def execute(self, w1, w2) -> dict: ...
```

Неизвестный ярлык → автоматически привязывается к первому свободному слоту. Известный → прямо в свою очередь.

## 5. Очереди входа и выбор кандидата

Каждый вход узла — именованный слот с FIFO-очередью. Когда на один слот пришло несколько пакетов, порядок выбора — FIFO по умолчанию, но правило сменное: это часть контроллера ввода.

## 6. Парные контроллеры — обратимая скобка вокруг тела

Контроллер ввода и контроллер вывода работают как обратимая пара. Вход применяет преобразование, выход — обратное. Тело между ними не знает, что его обернули.

```
collect → тело: execute → emit
```

**Контроллер ввода** (вневременной): решает готовность, выбирает кандидата из очереди, собирает входы в то, что увидит тело. `collect()` возвращает `Collected(data, mark)`:

- `data` — словарь именованных входов, передаётся в `execute(**data)`
- `mark` — непрозрачные метаданные, которые контроллер ввода хочет передать контроллеру вывода (например, порядковый индекс для переупорядочивания). Путешествует локальной переменной через такт; безопасен при конкурентном исполнении.

**Контроллер вывода** (вневременной): получает результат тела и `mark` в `emit(results, mark)`. Проставляет ярлыки источника и рассылает по узлам-получателям.

Применения пары:

- **Синхронизация.** Вход записывает порядковый индекс в `mark`; `OrderedInputController` придерживает пакеты, пришедшие не в порядке; `OrderedOutputController` использует `mark["idx"]`, чтобы выпускать результаты по возрастанию.
- **Объединение–разъединение.** Вход сворачивает несколько пакетов в один батч, тело обрабатывает цельным объектом, выход разрезает результат обратно.
- **Нормализация–денормализация.** Вход приводит значения к рабочему виду, тело считает, выход возвращает к исходному.

Обычному узлу контроллеры писать не надо — подставляются стандартные: `InputController` (FIFO, готов когда все слоты непусты) и `OutputController` (штампует ярлык источника, рассылает по связям).

## 7. Готовность и пробуждение

Приём данных идёт через узел, не напрямую в очередь. Получив пакет, контроллер ввода сразу отвечает, готов ли запуск:

```
Ready()          — запускай сейчас
Wait()           — не готов, ждать только новых данных
WaitUntil(T)     — не готов, но разбуди не позже T (дедлайн батча)
```

Засыпает и просыпается только исполнитель. Он собирает все дедлайны T и спит до «узел завершился или ближайший T». Контроллер ввода своего потока и таймера не имеет — он лишь называет дедлайн. Так батч по таймауту работает без отдельного потока и без активного ожидания.

## 8. Тело: синхронное, асинхронное, удалённое

Тело — это `execute`. Оно может быть:

- **синхронным** — обычная функция
- **асинхронным** — корутина; долгая или удалённая работа выражается через `await`

`Node.run(ctrl)` — всегда async-корутина. `SyncExecutor` оборачивает её в `asyncio.run()`; `AsyncExecutor` делает `await` напрямую с лимитом параллельности. Исполнителю всё равно, считает тело локально или ждёт сеть.

Ярлык источника, память узла и рычаги управления доступны внутри `execute` через context variables, привязанные на время такта:

```python
self.memory       # словарь узла, живёт между тактами
self.stop()       # мягкая остановка: текущие такты доигрывают, новые не стартуют
self.snapshot()   # снимок состояния исполнителя
self.to(name, v)  # создать TaskResult для fan-out
```

## 9. Результат задачи

`execute` возвращает одно из:

- `dict` — `{"имя_связи": value, ...}`; ключ `None` отправляет на выход графа
- `list[TaskResult]` — для fan-out или явного переопределения ярлыка
- `None` — нет выхода (fire-and-forget)

`TaskResult(value, target_node, label)` — явный узел-получатель и опциональный ярлык. Создаётся через `self.to(имя_связи, value, label=None)`.

## 10. Исполнитель

Один обход, два режима:

- кладёт результаты завершившихся узлов в очереди получателей (через их контроллер ввода), получает ответы готовности
- запускает все Ready-узлы
- засыпает до «узел завершился или ближайший дедлайн T»
- проснувшись, повторяет

**SyncExecutor** — `asyncio.run(node.run(ctrl))` на каждый узел; тела выполняются последовательно.
**AsyncExecutor** — `await node.run(ctrl)` через `asyncio.ensure_future`; все готовые тела запускаются конкурентно с лимитом `max_parallel`.

## 11. Управление через граф

Управляющий код — обычный узел графа. `self.stop()` и `self.snapshot()` доступны внутри любого `execute`. Остановка соседней ветви, сторож, лог — всё выражается узлами, без внешних слушателей.
