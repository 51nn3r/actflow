# actflow — спецификация

[English](SPEC.md)

Библиотека исполнения графов задач. Граф описывает, какие задачи выполняются и куда передают результаты; исполнитель прогоняет его, передавая данные между узлами. Поддерживает синхронный и асинхронный режим, в том числе удалённые и долгие задачи.

## 1. Базовая идея

Задача только считает и возвращает результат — куда его направить, задаётся связями графа, а не самой задачей. Решение о запуске принимает исполнитель. Это даёт ему полный контроль над порядком, параллельностью и остановкой.

Модель близка к функциональному стилю и к вычислительному графу: по рёбрам текут неизменяемые данные, узлы не делят общее изменяемое состояние.

## 2. Два уровня по доступу ко времени

Всё в узле делится на два уровня. Делёж — по тому, виден ли ход времени.

**Временной уровень — тело (execute).** Чистая функция одного такта: получило значения, вернуло значения. Прошлого и будущего не видит, между тактами ходить не может, состояние не хранит.

**Вневременной уровень — контроллеры.** Видят все очереди узла и накопленный выход, держат состояние между тактами. Сюда выносится всё, что требует памяти через время: готовность, выбор кандидата, порядок, сборка батча, разборка результата.

## 3. Пакет данных

По рёбрам течёт пакет — конверт над значением.

```
пакет = значение + ярлык типа
```

**Значение** — то, с чем работает тело. **Ярлык типа** — адрес слота: куда пакет ляжет в узле-получателе. Пакет неизменяем: один результат можно направить в несколько узлов без копий. Если задаче нужна изменяемая версия — она копирует сама, осознанно (важно для тензоров: лишних копий нет).

Метаданные (порядковый номер для синхронизации и пр.) по графу не ездят — они живут в состоянии контроллеров узла, который их завёл (см. раздел 6).

## 4. Маршрутизация по типу

Слот получателя определяется **ярлыком типа**, а не позицией прихода.

Узел по умолчанию помечает результат ярлыком своего типа: узел `a` выдаёт данные с ярлыком `A`. Узел `b`, объявленный принимающим `[A, C]`, кладёт пришедший `A` в очередь для `A`, пришедший `C` — в очередь для `C`.

Слоты узла заполняются типами при вставке в граф: по очереди, каждый тип — один раз; можно и явно. Если узел `d` хочет слать туда, где ждут `A`, он **меняет ярлык** результата на `A` перед отправкой — смена ярлыка есть выбор слота.

## 5. Очереди входа и выбор кандидата

Каждый вход узла — именованная очередь (по ярлыку типа). Когда на один слот пришло несколько пакетов, встаёт вопрос: какой брать следующим. По умолчанию FIFO, но правило выбора сменное — это часть контроллера ввода.

## 6. Парные контроллеры — обратимая скобка вокруг тела

Контроллер ввода и контроллер вывода работают как обратимая пара. Вход применяет преобразование, выход — обратное. Тело между ними не знает, что его обернули.

```
вход: преобразование → тело: вычисление → выход: обратное преобразование
```

**Контроллер ввода** (вневременной): решает готовность, выбирает кандидата из очереди, задаёт порядок, собирает входы в то, что увидит тело (например, сворачивает временные шаги в батч).

**Контроллер вывода** (вневременной): разбирает результат тела обратно по очередям-выходам (разрезает батч на шаги), проставляет ярлыки и адреса.

Вход и выход одного узла договариваются через **квитанцию**: пропуская пакеты в тело, вход возвращает «что запомнить» (например, порядковый номер); исполнитель проносит квитанцию до выхода и отдаёт её вместе с результатом тела. Квитанция едет с задачей и переживает переход через сеть — в отличие от общей памяти, она работает в распределённом режиме.

Применения пары:

- **Синхронизация.** Вход помнит порядковый номер пропущенного пакета, выход вешает его обратно и придерживает результаты, отдавая по возрастанию. Восстанавливает порядок, сбитый разной скоростью обработки.
- **Объединение–разъединение.** Вход сворачивает несколько пакетов в один батч, тело обрабатывает цельным объектом, выход разрезает результат обратно. Один прогон по батчу вместо N прогонов по шагам.
- **Нормализация–денормализация.** Вход приводит значения к рабочему виду (масштаб, форма), тело считает в нём, выход возвращает к исходному виду.

Обычному узлу контроллеры писать не надо — подставляются стандартные: вход — FIFO «по пакету из каждой очереди, готов когда все есть»; выход — пометить результат ярлыком-типом узла и разложить по связям.

## 7. Готовность и пробуждение

Приём данных идёт через узел, не напрямую в очередь. Получив пакет, контроллер ввода сразу отвечает, готов ли запуск. Ответ — один из трёх:

```
READY           — запускай сейчас
WAIT            — не готов, ждать только новых данных
WAIT_UNTIL(T)   — не готов, но разбуди не позже момента T (дедлайн батча)
```

Засыпает и просыпается только исполнитель. Он собирает все дедлайны T и спит до «узел завершился или ближайший T». Контроллер ввода своего потока и таймера не имеет — он лишь называет дедлайн. Так батч по таймауту работает без отдельного потока и без активного ожидания.

## 8. Тело: синхронное, асинхронное, удалённое

Тело — это `execute`. Оно может быть:

- **синхронным** — обычная функция, исполнитель ждёт её на месте
- **асинхронным** — корутина; долгая или удалённая работа (redis, другой узел) выражается через `await`, исполнитель в это время обрабатывает других

Исполнителю всё равно, считает тело локально или ждёт сеть: в async-режиме он вызвал `execute` всех готовых узлов и уснул; первая завершившаяся корутина его разбудила. Отдельного механизма возобновления нет.

## 9. Результат задачи

Тело возвращает список адресованных результатов:

```
TaskResult = (данные, узел-получатель)
```

Обычно один результат и один получатель; список нужен для ветвления и fan-out. Слот внутри получателя выбирается по ярлыку типа (раздел 4). Вывод наружу графа — это отдельная терминальная задача, а не особый результат.

## 10. Исполнитель

Один обход, два режима:

- кладёт результаты завершившихся узлов в очереди получателей (через их контроллер ввода), получает ответы готовности
- запускает все READY-узлы
- засыпает до «узел завершился или ближайший дедлайн T»
- проснувшись, повторяет

**Синхронный** ждёт каждое тело на месте (готовые узлы — последовательно). **Асинхронный** запускает готовые тела разом и спит на их завершении, с лимитом параллельности; долгие и удалённые тела прозрачны.

Спящий всегда один — исполнитель — на `asyncio.wait` с таймаутом (async) или `time.sleep` (sync). Активного ожидания нет: при дедлайнах в доли секунды оно сожгло бы ядро впустую и в async задушило бы ожидающие корутины.

## 11. Управление через граф

Управляющий код — обычный узел графа. Через контекст ему доступны рычаги исполнителя: остановить (мягко: текущие запуски доигрывают, новые не стартуют) и снимок состояния для логирования. Остановка соседней ветви, сторож, лог — всё выражается узлами, без внешних слушателей.
